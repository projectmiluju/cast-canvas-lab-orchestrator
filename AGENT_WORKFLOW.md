# Agent Workflow

CastCanvas Lab에서 `Claude`, `Gemini`, `Codex`를 함께 사용할 때의 기본 역할 분담 문서다.

## 역할 분담

### Claude

- 실제 코드 구현
- 이슈를 읽고 구현 범위 파악
- 해당 레포에서 브랜치 생성
- 테스트, 빌드, 리팩터링
- 기능 단위 작업 완료

### Gemini

- TIL 작성
- 회고, 릴리즈 노트, 설명형 문서
- 장문 서술형 정리
- 작업 맥락을 사람이 읽기 쉬운 문장으로 재구성

### Codex

- 커밋 메시지 생성
- PR 본문 초안 생성
- 다음 이슈 초안 생성
- 에픽 생성/갱신
- GitHub Project, 이슈 계층, 위키, 템플릿 운영
- 크로스 레포 다음 작업 판단

## 기본 원칙

- 구현은 각 레포 안에서 진행한다.
- 크로스 레포 계획과 에픽 관리는 `cast-canvas-lab-orchestrator`에서 진행한다.
- 문서형 산출물과 운영 구조는 구현과 분리해 관리한다.

## 권장 작업 흐름

### 1. 작업 시작

1. Claude가 해당 레포 이슈를 읽는다.
2. Claude가 해당 레포 안에서 브랜치를 만든다.
3. Claude가 구현, 테스트, 리팩터링을 진행한다.

## 2. 작업 종료 정리

1. 같은 레포 안에서 Codex를 실행한다.
2. `$commit-message`로 커밋 메시지를 만든다.
3. `$pr-and-next-issue`로 PR 초안과 다음 이슈 초안을 만든다.
4. 필요하면 GitHub 이슈나 PR 본문에 반영한다.

## 3. 크로스 레포 다음 작업 판단

1. `cast-canvas-lab-orchestrator`로 이동한다.
2. Codex를 실행한다.
3. `$cross-repo-planner`를 사용한다.
4. 에픽 재사용 또는 신규 에픽 생성 여부를 결정한다.
5. 다음 작업이 어느 레포에 가야 하는지 결정한다.
6. 필요하면 해당 레포 이슈를 만든다.

## 어떤 작업을 누구에게 줄까

### 이슈를 읽고 어떤 작업을 진행할지 판단

기본적으로는 `Claude`에게 준다.

이유:

- 바로 구현으로 이어지는 해석이 필요하다.
- 현재 레포 코드와 이슈를 함께 보고 작업 범위를 좁히기 좋다.
- 브랜치 생성과 구현까지 자연스럽게 이어진다.

예외:

- 여러 레포 상태를 같이 봐야 하면 `Codex`에게 준다.
- 이 경우 `orchestrator`에서 `$cross-repo-planner`를 사용한다.

### 브랜치 생성

`Claude`에게 준다.

이유:

- 브랜치는 구현 시작과 함께 만드는 게 가장 자연스럽다.
- 실제 작업 단위와 브랜치 경계가 같이 가야 한다.
- 각 레포별 Git 저장소 안에서 바로 처리하는 편이 안전하다.

### TIL, 회고, 릴리즈 노트

`Gemini`에게 준다.

### 커밋 메시지, PR 초안, 다음 이슈, 에픽 정리

`Codex`에게 준다.
